# API reference

REST + JSON for everything except the chat endpoint, which is Server-Sent
Events. All examples assume the `chat.localhost` entry point in dev — swap
in your public hostname for a deployment.

> An interactive Swagger UI is auto-generated by FastAPI at
> [`/api/docs`](https://chat.localhost/api/docs) (`/docs` upstream). The
> raw OpenAPI schema is at `/api/openapi.json` and is committed to
> `backend/openapi.json` as the contract the frontend's TS types are
> generated from.

## Conventions

- All non-public endpoints expect `Authorization: Bearer <access_token>`.
  The bearer token is a 15-minute JWT obtained from `/auth/login`. See
  [`auth.md`](auth.md) for the full handshake.
- Bodies are JSON unless noted.
- Errors follow FastAPI's default shape: `{"detail": "<message>"}` with the
  appropriate HTTP status.
- Through Caddy, every backend route is reachable under `/api/...` thanks
  to the same-origin split (the `/api/` prefix is stripped by Caddy
  before hitting FastAPI). The examples below show both the raw FastAPI
  path (in the verb line) and the through-Caddy URL (in the curl).

## Auth

### `POST /auth/register`

Public. Creates an account.

```bash
curl -X POST https://chat.localhost/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","email":"alice@example.com","password":"a-good-password"}'
```

`201 Created` → `UserPublic` JSON. `409 Conflict` if username or email
already taken. Username is `^[a-zA-Z0-9_-]+$`, 3–50 chars; password 8–128
chars; email validated by Pydantic.

### `POST /auth/login`

Public.

```bash
curl -X POST https://chat.localhost/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"a-good-password"}'
```

`200 OK` →

```json
{
  "access_token": "<JWT>",
  "refresh_token": "<opaque>",
  "token_type": "bearer",
  "access_expires_at": "2026-04-23T10:30:00Z",
  "refresh_expires_at": "2026-05-23T10:15:00Z"
}
```

`401` on bad credentials.

### `POST /auth/refresh`

Public. Rotates: the old refresh is revoked, a new pair is issued.

```bash
curl -X POST https://chat.localhost/api/auth/refresh \
  -H "Content-Type: application/json" \
  -d '{"refresh_token":"<opaque>"}'
```

`401` if the refresh is unknown, expired, or already revoked.

### `POST /auth/logout`

Public. Idempotent — always `204` even if the token was already revoked.

```bash
curl -X POST https://chat.localhost/api/auth/logout \
  -H "Content-Type: application/json" \
  -d '{"refresh_token":"<opaque>"}'
```

### `GET /auth/me`

Authenticated.

```bash
curl https://chat.localhost/api/auth/me -H "Authorization: Bearer $TOKEN"
```

→ `UserPublic` (id, username, email, avatar_url, locality, time_format,
temperature_unit, defaults, default_persona_id, created_at).

### `PATCH /auth/me`

Authenticated. Partial update — every field is optional, only those
present in the body are touched.

```bash
curl -X PATCH https://chat.localhost/api/auth/me \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"locality":"Paris","time_format":"24h","defaults":{"temperature":0.6}}'
```

`default_persona_id` uses `null` to detach (`{"default_persona_id":null}`)
vs omission to leave unchanged. Same trick for `model` on `PATCH /v1/sessions/{id}`.

## Sessions

### `GET /v1/sessions`

```bash
curl "https://chat.localhost/api/v1/sessions?archived=false&limit=50&offset=0" \
  -H "Authorization: Bearer $TOKEN"
```

→ `SessionPublic[]`, newest `updated_at` first.

### `POST /v1/sessions`

```bash
curl -X POST https://chat.localhost/api/v1/sessions \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"title":"New conversation","persona_id":3,"model":null,"overrides":{"temperature":0.4}}'
```

`201 Created` → `SessionPublic`. If `persona_id` is omitted entirely, the
backend falls back to `users.default_persona_id`. Pass `null` explicitly
to mean "no persona".

### `GET /v1/sessions/{id}`

→ `SessionDetail` = `SessionPublic` + `messages: MessagePublic[]`. Tool
results are folded into the `tool_calls[].result` of the assistant
message that requested them, and consecutive multi-round assistant
messages from the same turn are merged into one bubble.

### `PATCH /v1/sessions/{id}`

```bash
curl -X PATCH https://chat.localhost/api/v1/sessions/$SID \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"title":"vLLM tuning notes","archived":false,"overrides":{"top_p":0.9}}'
```

### `DELETE /v1/sessions/{id}`

`204 No Content`. Cascades to messages.

### `GET /v1/sessions/{id}/messages`

→ `MessagePublic[]` (same shape as the `messages` field of `SessionDetail`).

### `POST /v1/sessions/{id}/messages` — SSE chat

The streaming endpoint. Request body is the new user message:

```bash
curl -N -X POST https://chat.localhost/api/v1/sessions/$SID/messages \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{"content":"What is the weather in Paris?"}'
```

Response is `text/event-stream`. Event types:

| Event | Data shape | When |
|---|---|---|
| `message.start` | `{ "round": 0 }` | Each new round (rounds > 0 are post-tool continuations) |
| `reasoning.delta` | `{ "text": "..." }` | Each chunk of the model's `<think>…</think>` block |
| `message.delta` | `{ "text": "..." }` | Each chunk of the visible reply |
| `tool.call` | `{ "id": "call_xx", "name": "web_search", "arguments": { … } }` | Model decided to call a tool; arguments are already parsed |
| `tool.result` | `{ "id": "call_xx", "result": { … }, "duration_ms": 420 }` | Backend ran the tool and is sending the result before the continuation round |
| `message.done` | `{ "message_id": "...", "finish_reason": "stop", "usage": {...}, "timings": {...} }` | Final round finished |
| `error` | `{ "message": "..." }` | Stream-level errors |

The full pipeline (multi-round tool calls, persistence, reasoning
extraction) is documented in [`chat.md`](chat.md).

## Personas

```
GET    /v1/personas               list (own + built-ins)
GET    /v1/personas/{id}          fetch one
POST   /v1/personas               create
PATCH  /v1/personas/{id}          update (403 on built-ins)
DELETE /v1/personas/{id}          delete (403 on built-ins, 204 on success)
```

Body for `POST` / `PATCH`:

```json
{ "name": "Sarcastic mentor", "system_prompt": "You are…", "params": {} }
```

`name` 1–100 chars, `system_prompt` 1–20 000 chars (markdown OK — the UI
renders it).

## Tools

```
GET   /v1/tools                  list available tools (with enabled state)
PUT   /v1/tools/{name}           toggle one for the current user
```

```bash
curl -X PUT https://chat.localhost/api/v1/tools/web_search \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"enabled": true}'
```

The persisted state is one row in `settings` keyed by `tools.enabled`
holding a JSON array of names. Defaults come from each tool's
`enabled_by_default` flag.

## Runtime config

```
GET /v1/config
```

Returns `{ "version": "...", "model": "...", "model_source": "vllm" | "config" }`.
The frontend terminal footer reads this to display the current model
name. `model_source = "vllm"` means the value was fetched live from
vLLM's `/v1/models`; `"config"` means vLLM was unreachable and we fell
back to the `VLLM_SERVED_MODEL_NAME` env var.

## Health

```
GET /health
```

Public, used by the Docker healthcheck and any external uptime monitor.
Returns `{ "status": "ok", "version": "..." }`.

## OpenAPI → TypeScript

`backend/openapi.json` is the single source of truth. The frontend types
are regenerated from it via:

```bash
make gen-api-types
```

`make check-api-types` is the CI hook — it regenerates and `git diff
--exit-code`s. Drift is therefore impossible to land on `main`.

The generated types live in `frontend/src/types/api-generated.ts`;
domain-specific refinements (narrower `MessagePublic`, refined `ToolCall`
shape, the SSE event types) live in `frontend/src/types/api.ts`.
